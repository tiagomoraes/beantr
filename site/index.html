01 Get started

Paste one prompt. Then just talk to your agent.

The simplest way to install Beantr is to let your agent do it: paste the prompt below and it fetches the pack, adds the skill, and creates your ledger. Any file-capable assistant can run it.

STEP 1 Install the skill

Paste this to your assistant

Install Beantr so you can track my coffee. Run curl -fsSL https://beantr.tiagomoraes.cloud/install | bash. It adds the Beantr skill and creates a Markdown coffee ledger at ~/beantr. Then confirm you can read it.

Works with any agent that can run shell commands: Claude Code, Hermes, OpenCode, or OpenClaw. It fetches the pack, adds the skill, and creates your ledger.

Rather run it yourself?

curl -fsSL https://beantr.tiagomoraes.cloud/install | bash

In Claude Code, install it as a plugin instead:

/plugin marketplace add tiagomoraes/beantr
/plugin install beantr@beantr

Installing for a specific agent?

Hermes
Installs as a native skill, loaded directly.
./installers/install.sh hermes ~/beantr
Claude Code
Copies the skill and adds a CLAUDE.md reminder of the ledger path.
./installers/install.sh claude-code ~/beantr
OpenCode
Copies the skill and appends an AGENTS.md block.
./installers/install.sh opencode ~/beantr
OpenClaw
Copies the skill into OpenClaw's global skills directory.
./installers/install.sh openclaw ~/beantr
Claude Cowork
Packages a skill zip to upload in Settings → Customize.
./installers/install.sh cowork ~/beantr
Generic file-capable assistant
One instruction file. No skill support required.
./installers/install.sh generic ~/beantr

Prefer to inspect the pack first, or run it in CI? Clone the repo, download the tarball, or read the skill file directly.

02 The model

Three actors. One contract.

You chat / voice Agent read / write ~/beantr/*.md

You talk to your agent the way you always do, by chat or by voice. The agent reads and writes a folder of Markdown on your machine. Beantr is what's written on those files: a contract for what goes where, and what it means.

03 Three kinds of truth

Every file answers exactly one question.

Live state
What is true right now?
current.md files are mutated in place: beans on hand, gear in use, the recipe you'd reach for today.
Evidence
What actually happened?
Monthly logs are append-only. Every brew, every restock, every correction: preserved, never rewritten.
Distillation
What should you do next?
insights/current.md updates only when evidence is strong enough to be useful. Never generic theory.
beantr/
├── README.md
├── beans/current.md
│   └── history/2026-06.md
├── gear/current.md
│   └── history/2026-06.md
├── recipes/current.md

└── insights/current.md

05 Five rules

What Beantr will not compromise on.

01
The filesystem is the API.
Agents use ordinary file tools: no HTTP calls, no MCP tools, no database credentials.
02
Current state beats inferred state.
A current.md file answers what's true now. The agent shouldn't replay history to find out.
03
Logs stay append-only.
History is evidence, not mutable state; session and history files only ever grow.
04
Recommendations must cite evidence.
Good advice points to recent sessions or stable preferences, never generic theory.
05
Humans can always edit the files.
Beantr never requires a special UI: open any file, in any editor, anytime.

No dashboard. No gradients. Just files, kept well.
